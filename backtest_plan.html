Backtest Plan

백테스트 구현 계획

현재 앱의 무한매수법 V4 계산 로직을 과거 일봉 데이터로 검증하기 위한 데이터 수집, 체결 가정, 시뮬레이션 흐름, 결과 화면 설계입니다.

데이터 체결 가정 시뮬레이션 결과 지표 UI 구현 MVP
1순위 데이터

Yahoo Finance 일봉 CSV. 먼저 내려받아 고정 파일로 사용합니다.

핵심 가정

LOC/LIMIT 체결은 당일 OHLC 범위로 판정합니다.

초기 상태

첫 거래일 종가로 1회분을 이미 매수한 상태에서 시작합니다.

우선 구현

TQQQ/SOXL 단일 전략, 수수료 없는 라운드별 결과부터 만듭니다.

목표

  • 현재 앱의 무한매수법 V4 계산 로직이 과거 일봉 데이터에서 어떻게 동작했는지 검증합니다.
  • 라운드별 수익률, 누적 수익률, 최대 낙폭, 현금 소진/리버스모드 기간, 체결 내역을 확인합니다.
  • 실제 주문 자동화가 아니라 전략 검증용 도구로만 사용합니다.

가격 데이터 형식

파일은 먼저 고정 CSV로 관리합니다.

data/prices/TQQQ.csv
data/prices/SOXL.csv

CSV 컬럼은 다음처럼 둡니다.

date,open,high,low,close,adj_close,volume
2024-06-27,73.12,74.40,72.80,73.95,73.95,123456789
  • 날짜는 미국 장 기준 YYYY-MM-DD입니다.
  • 가격은 split-adjusted 값이어야 합니다.
  • 누락 거래일은 채우지 않고 실제 거래일 행만 사용합니다.
  • 로딩 시 날짜 오름차순 정렬, 중복 날짜는 실패 처리합니다.

RAM 데이터 처리

앱 계산에서 RAM은 SOXL과 같은 20% 기준을 쓰지만, 백테스트 가격 데이터는 별개입니다.

  • RAM이 실제 거래 티커라면 별도 가격 CSV가 필요합니다.
  • 공개 일봉 데이터 출처가 불명확하면 MVP에서는 RAM 직접 백테스트를 제외합니다.
  • SOXL 가격 데이터에 매핑하는 옵션을 둘 수 있지만, 결과에 RAM 계산식 + SOXL 가격 데이터 사용이라고 명확히 표시해야 합니다.

체결 가정

LOC/MOC 체결을 완벽히 재현할 수 없으므로 MVP에서는 보수적인 OHLC 규칙을 고정합니다.

주문체결 조건체결가
LOC 매수low <= orderPrice <= high주문 가격
LOC 매도low <= orderPrice <= high주문 가격
LIMIT 매도high >= limitPricelimitPrice
MOC 매도당일 장 종료종가 close
리버스 MANUAL 매수종가가 5일 평균보다 낮음종가 close
리버스 MANUAL 매도종가가 5일 평균보다 높음종가 close

초기 상태 가정

백테스트는 사용자가 이미 첫 매수를 하고 전략을 시작한다는 현재 앱 사용 흐름에 맞춥니다.

cash_balance = principal - 첫 매수 금액
position_qty = floor((principal / split_count) / startClose)
avg_price = startClose
t_value = 1
mode = normal

첫 매수 수량이 0이면 원금 부족으로 백테스트 시작을 실패 처리합니다.

시뮬레이션 흐름

  1. CSV 가격 데이터를 로드합니다.
  2. 시작일과 종료일 사이 거래일만 남깁니다.
  3. 첫 거래일 종가로 초기 매수를 만듭니다.
  4. 다음 거래일부터 매일 반복합니다.
  5. 현재 StrategyState로 calculateNormalPlan 또는 calculateReversePlan을 호출합니다.
  6. 생성된 주문 가이드를 당일 OHLC 체결 가정으로 평가합니다.
  7. 체결된 주문만 상태에 반영합니다.
  8. T값은 현재 applyTEffect 규칙을 사용합니다.
  9. 보유수량이 0이면 라운드를 종료하고 결과를 저장합니다.
  10. 복리/단리 설정에 따라 새 라운드 상태를 초기화합니다.
  11. 종료일까지 반복합니다.

상태 반영 규칙

매수 체결

  • cash_balance -= quantity * price
  • position_qty += quantity
  • avg_price = 총 매입 원가 / position_qty
  • T값은 buy_full 또는 buy_half를 적용합니다.

매도 체결

  • cash_balance += quantity * price
  • position_qty -= quantity
  • 남은 수량이 있으면 평단은 유지합니다.
  • 남은 수량이 0이면 새 라운드를 시작합니다.

같은 날 여러 주문이 체결되면 MVP에서는 매도 주문을 먼저 반영하고 매수 주문을 반영합니다.

데이터 다운로드 스크립트

MVP에서는 앱 안에서 외부 API를 호출하지 말고 스크립트로 CSV를 만듭니다.

scripts/download-prices.ts

npm run prices -- TQQQ 2010-01-01 2026-12-31
npm run prices -- SOXL 2010-01-01 2026-12-31

현재 repo는 npm 기반이므로 앱 코드와 통합하려면 Node 스크립트가 가장 단순합니다.

Yahoo Chart API 예시

https://query1.finance.yahoo.com/v8/finance/chart/TQQQ?period1=1262304000&period2=1798675200&interval=1d&events=history%7Csplit%7Cdiv
  • period1, period2는 Unix timestamp 초 단위입니다.
  • 응답의 adjclose와 split 이벤트를 확인합니다.
  • OHLC 조정은 adj_close / close 비율을 각 날짜의 open/high/low/close에 곱합니다.
  • close가 0이거나 누락된 날짜는 제거합니다.

검증 방법

  1. 10거래일짜리 작은 CSV fixture를 손으로 만듭니다.
  2. 첫 매수, 전반전 매수, 후반전 매수, 쿼터매도, 최종매도, 리버스모드 진입을 각각 강제로 발생시키는 fixture를 만듭니다.
  3. 각 fixture에서 체결 수량, 현금, 보유수량, T값이 예상과 같은지 확인합니다.
  4. 실제 TQQQ 1년 구간으로 smoke test를 돌려 결과가 음수/NaN 없이 생성되는지 확인합니다.
  5. 같은 입력을 두 번 실행했을 때 결과가 완전히 동일해야 합니다.

주요 리스크

  • LOC 주문은 실제 체결 규칙이 복잡해서 OHLC만으로 정확히 재현할 수 없습니다.
  • 무료 데이터는 분할/배당 조정 방식이 바뀔 수 있습니다.
  • SOXL/TQQQ 같은 레버리지 ETF는 장기 데이터에서 분할이 많아 조정 가격 검증이 중요합니다.
  • 리버스모드의 수동 판단은 앱 사용자의 실제 행동과 달라질 수 있습니다.
  • 세금, 수수료, 슬리피지는 MVP에서 제외하면 실제 수익보다 낙관적으로 나옵니다.
